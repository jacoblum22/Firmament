## Priority 4: Fix Infrastructure Issues

### 4a. docker-compose.yml service name inconsistency
In `backend/docker-compose.yml`, the API service is defined as `firmament-api` but Nginx `depends_on` references `studymate-api`. Fix:

```yaml
# Change the nginx depends_on to match:
depends_on:
  - firmament-api    # was: studymate-api
```

### 4b. Dual pytest.ini
Root `pytest.ini` and `backend/pytest.ini` both configure pytest.  
Root `pytest.ini` says `testpaths = backend/tests` — this is the whole-project runner.  
`backend/pytest.ini` says `testpaths = tests` — this is for running from inside `backend/`.  
Both are legitimate for different use cases. Add a comment to each noting when to use it:

Root `pytest.ini` → for running from project root (`pytest` from `StudyMate-v2/`)  
`backend/pytest.ini` → for running from `backend/` directory

### 4c. requirements.in is misleading
`requirements.in` only has 8 packages but `requirements.txt` has ~15+ including boto3, google-auth, PyJWT, redis, etc. Either:
- Update `requirements.in` to be the true source of truth, then `pip-compile` to regenerate `requirements.txt`
- Or remove `requirements.in` and the `Makefile` `update-deps` target that depends on it
- The `requirements-fast.txt` actually serves the "minimal" purpose better

### 4d. Dockerfile COPY order issue
In `Dockerfile`:
```dockerfile
COPY --chown=appuser:appuser requirements.txt .
RUN pip install ...
# ...then later:
RUN adduser --disabled-password --gecos '' appuser   # ← user created AFTER chown
```
The `--chown=appuser:appuser` in the first COPY will fail because `appuser` doesn't exist yet.
Fix: create the user BEFORE the COPY lines, or remove the `--chown` from the requirements COPY.

### 4e. Redundant server launch scripts
Three ways to start the server: `start.py`, `start.ps1`, `deploy.py`.  
Recommendation:
- Keep `start.py` as the canonical Python entry point
- Keep `start.ps1` as the Windows convenience wrapper (it delegates to `start.py`)
- Rename `deploy.py` to `predeploy_check.py` or add a `--check-only` flag to make its role clearer vs. just launching the server
